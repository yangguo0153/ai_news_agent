# 媒体评估管理系统 - 汇报与初期规划

## 目标
基于既有SaaS平台能力和客户新增需求，形成一版可对齐的实施节点图与交付节奏说明，用于客户汇报与立项确认。

## 范围
- 输入材料：`交付节点流程图.pptx`（第一轮沟通后版本）
- 新增需求：
  - 媒体标签自定义标注（手动标注/取消）
  - 媒体人动态实时更新展示（在前一项开发后）

## Phase
| 阶段 | 状态 | 说明 |
|---|---|---|
| Phase 1: 资料解析与约束识别 | completed | 提取既有流程图时间轴与关键阶段 |
| Phase 2: 新需求影响评估 | completed | 评估新增功能对开发、测试、上线的影响 |
| Phase 3: 初版计划输出（用于汇报） | completed | 已形成并落地到PPT成品（6周稳妥版） |
| Phase 4: 与你共创细化计划 | in_progress | 根据你反馈继续细化资源、职责与验收口径 |

## 当前假设
1. 新增需求的“2周上线”指“开发完成并通过基础联调后可发布到生产”，不含需求反复变更时间。
2. 第二项需求需在第一项“开发完”后开始，默认是串行而非并行。
3. 仍沿用既有成熟平台，主要是定制改造而非从0开发。

## 待确认
1. 客户期望“实时更新”是秒级推送、分钟级轮询，还是日级同步。
2. 媒体标签是否需要权限控制（谁可标注/谁可撤销）。
3. 是否需要历史操作审计（谁在何时改了什么标签）。
4. 是否已确定UAT参与人和验收窗口。

## 风险与应对
| 风险 | 影响 | 应对 |
|---|---|---|
| “实时”定义不清 | 开发量和工期偏差 | 先锁定刷新频率和数据源，再估时 |
| 标签权限与审计后置提出 | 返工与延期 | 在需求冻结前把权限/审计作为必答项 |
| 客户反馈轮次过多 | 压缩测试和上线窗口 | 设定每轮反馈截止日，超出进入下一迭代 |

## 交付物（本轮）
- 调研摘要（已写入 `findings.md`）
- 初版排期建议（用于汇报讲解）
- 下一步共创问题清单（用于planning阶段细化）

## 已确认决策（2026-02-24）
1. “实时更新”按天同步
2. 标签功能包含权限控制与审计
3. UAT节奏：每周5次，每次1小时
4. 对外承诺：稳妥6周
